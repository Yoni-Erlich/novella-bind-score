1. Data, cleaning & what to watch for

881 submissions (one per submissionId), 14.8% bind rate → imbalanced, so we use ranking metrics, not accuracy. ~17k events of 3 types (EMAIL_INBOUND / OUTBOUND / QUOTE_RECEIVED). A row is one (submission, t) pair, scored only while the submission is still open at t.

Cleaning policy (2 drops, leakage-safe):
  • 530 exact-duplicate events (same sub + second + type + chars + attach) — concentrated in high-activity subs, so they inflate the volume features that carry our signal.
  • 127 impossible pre-creation events (dated >1 day before the submission exists) — counted at every t (incl. t=0), falsely inflating day-0 activity.
Watch out for:
  • Leakage is the central risk: feature(sub, t) may use only events with eventDate ≤ createdDate + t, and never resolvedDate/label.
  • Domain quirk, NOT leakage: OUTBOUND / quotes can precede the first INBOUND — intake happens via createdDate (off-channel), carriers quote off-channel. Kept.
  • Char outliers (max 43k, p99 2.3k): kept, tamed by log1p — not deleted.
  • Censoring: by t=30 >50% have resolved; only score open-at-t submissions.

2. Features chosen (Task 1)

The signal is simple: activity volume + quote presence + customer track record. Normalized ratios, attachments, char-distribution stats, response-latency, velocity and calendar all add nothing beyond raw volume (see Appendix A).

The 4 predictive features + t (a context variable that lets one pooled model adapt across snapshots). This table is the one-stop summary of what we did with each feature — what it captures, how it's built (leakage-safe), and its preprocessing:

Standardize = z-score z=(x−μ)/σ (μ/σ from train only), inside the pipeline Impute(median) → StandardScaler → Logistic.

Why agent_bind_rate matters most: agentEmail is Novella's customer (the retail agent), not the broker. A customer's prior close-rate is known at t=0 — the one strong signal available on day 0 (for returning customers), when nothing has happened on the submission yet. Adding it lifted the day-0 ranking from a coin-flip (ROC-AUC ≈0.50) to 0.76.
Note: 0.50 is the AUC floor (random ranker), not a bind rate. The 15% base rate is the agent_bind_rate feature value a first-timer gets (see the cold-start callout in §3.3).
Building agent_bind_rate — the no-history dilemma & how it updates. Two problems: a first-time agent has no history (no rate to compute), and an agent with a single prior deal that happened to bind would naively read as 100%. Solution — empirical-Bayes smoothing: blend the agent's own record with the global average, weighted by how much history they have:
rate = (prior_binds + α·g) / (prior_n + α)   α = 5 pseudo-deals, g = base rate (15%)
  • No history (prior_n = 0) → rate = g: a new customer is "assumed average until proven otherwise."
  • Self-updating: each resolved deal grows prior_n, sliding the estimate from g toward the agent's true close-rate; one fluke deal is diluted by the 5 pseudo-deals.
  • Leakage-safe: uses only the agent's deals resolved before this submission's createdDate; g is the train base rate (recomputed after the split).

Worked example (g = 15%, α = 5): new agent 0/0 → 15%  ·  one win 1/1 → (1 + 5·0.15)/6 = 29% (not 100%)  ·  established 8/20 → (8 + 5·0.15)/25 = 35% (≈ their true 40%). Thin history stays near 15%; lots of history trusts the agent.

2.2 Explainability — what each feature means

Plain-English read of each coefficient. Four are exactly what you'd expect; outbound_chars's sign is a transform artifact and should not be interpreted (see the box).

featuresignplain meaning
agent_bind_rate+the customer's past close-rate — trusted repeat customers bind more. Intuitive.
has_quote_by_t+a carrier quote arrived — the funnel gate; big jump in odds. Intuitive.
n_inbound_by_t+customer replies = engagement → mildly higher odds. Intuitive.
t≈0snapshot time — a context variable so one pooled model serves t=0/7/30; no standalone meaning.
outbound_chars_log−broker effort — sign is not interpretable (an artifact). See below.
outbound_chars's negative coefficient has no real meaning — it's a collinearity artifact. The feature is redundant (kept, but not load-bearing); only its sign is uninterpretable.
  • The sign isn't robust. Refitting with only the outbound encoding changed, its coefficient is +0.354 (raw) · -0.139 (log1p, current) · +0.110 (quantile bins) — it flips sign with the transform, while held-out AUC is identical (~0.744). The sign is decided by an arbitrary encoding choice, not by the data.
  • Why — collinearity. A logistic coefficient is a partial effect (the feature's contribution given the others). Outbound is strongly collinear with has_quote (r=+0.56) and n_inbound (r=+0.53) — the features that carry the real signal — while its own marginal correlation with binding is only +0.04. When predictors are collinear the model splits the shared signal between them somewhat arbitrarily, so outbound's residual coefficient sits at the zero-crossing and a nonlinear transform tips it either way. The direction is simply not identified.
  • Does it "earn its place"? The data can't say — the decision is noise-dominated. See §2.3: with ~130 positive submissions and features correlated at 0.5+, every feature set sits within cross-validation noise of every other. We keep the full informative set because the held-out test — our one out-of-sample arbiter — mildly favors it, and including it costs nothing. (Dropping it moves held-out AUC by ~0.01.)
Bottom line: outbound is redundant and its coefficient sign is uninterpretable, but the which-feature choice among these correlated predictors is below the noise floor — so we keep the full set and don't over-interpret. Full record: decisions.md; investigation: notebooks/03_outbound_coefficient.ipynb.

2.3 Did outbound "earn its place"? — exhaustive best-subset selection

With only 4 real predictors (t is the structural snapshot index, always in) there are just 24−1 = 15 feature subsets — so we don't approximate with greedy forward selection, we run them all (the optimal version of what greedy approximates). Each is scored by grouped 5-fold CV precision@20, averaged over 5 seeds, on the training set only; the winner is confirmed once on the held-out test.

outbound+t (worst)✓0.1970.541
Why CV can't choose: the measurement error is bigger than the spread.
  • The spread is 0.005; one CV run's own error is ±0.009. Re-running 5-fold CV 20 times on the same fixed feature set — only the random fold split changes — precision@20 lands anywhere from 0.230 to 0.263 (std ±0.009). The entire gap between the best and worst of the top subsets is only 0.005, well inside that wobble. So which feature set "wins" is decided by the random fold split, not by the features — exactly why a single-seed run once favored outbound and multi-seed averaging erased it. With only ~130 positive submissions, CV folding simply has no resolution at the 0.005 level.
  • Single-feature reality. outbound is the weakest candidate: lowest marginal correlation with binding (+0.04) and the worst single feature by CV precision@20 (0.197, vs the full set's 0.248). Greedy selection seeded from the strongest feature agrees — it stops at agent+quote and never adds outbound. (The earlier "outbound is the best single feature" claim was a single-seed artifact; multi-seed averaging dissolves it.)
  • The arbiter. Since CV can't decide, we defer to the one out-of-sample check. On the held-out test the full set edges the CV "winner" on both ROC-AUC (0.744 vs 0.733) and precision@20% (33% vs 30% — a ≈3-submission gap in the top bucket). It mildly favors keeping the full set, and including it costs nothing — so we do.

3. Modeling (Task 3)

3.1 Preprocessing & validation

(a) Clean the data — apply the 2 drops above in-pipeline (never mutate data/). (b) Build features per (submission, t), leakage-safe — per-feature build & transforms are in the §2 table. Inside the pipeline Impute(median) → StandardScaler → Logistic, every feature is standardized (z-score, μ/σ from train only) → all 5 end mean 0, sd 1.

Why standardize: L2 penalizes coefficients equally regardless of units, and the optimizer converges faster on a well-scaled loss surface; it also makes the coefficients comparable for the §2.1 ranking.

(c) Validation strategy — three-way, no peeking:
  • Temporal split: train = earliest ~70% of submissions (by createdDate); held-out test = latest ~30% (578 rows). Train-on-past / test-on-future is the honest setup for a repeat-customer signal.
  • Validation = grouped CV inside the training set: every choice (feature set, the log1p transform, regularization) is picked by 5-fold StratifiedGroupKFold, grouped by submissionId so a submission's t=0/7/30 rows never straddle a fold — no within-submission leakage. We do not carve a static validation set: with only 130 positives, CV reuses the data more efficiently.
  • The held-out test is touched once — for the final numbers below; it never informs a modeling decision.

3.2 How we measure

Precision@k is primary. Brokers work a ranked queue top-down, so the operational question is "of the top k% I'm told to work, what share actually bind?" — exactly precision@top-k (k = 20/30/50%). Lift = precision@k ÷ the random level = how much better than working at random. (A no-skill model can't rank, so its top-k% is just a random sample → its precision is the same at every k; that random level is what we divide by, so lift reads directly as "× over random.")
Secondary: PR-AUC (precision/recall — the right summary under imbalance) and ROC-AUC (familiar, but optimistic on imbalanced data because it rewards ranking the abundant non-binders — so we don't lead with it).
The no-skill floors differ by metric: a random ranker scores ROC-AUC 0.50; its precision@k / PR-AUC sit at the dataset prevalence (≈15%). 0.50 is a ranking floor, not a rate — that's why the AUC floor is 0.50, not 15%.

3.4 Why not just the customer bind-rate?

A fair question, since agent_bind_rate alone nearly matches the full model overall. But that overall number hides a split — segment the test by whether the customer has history:

segmentnagent_bind_rate alonefull model
overall5780.7420.744
returning customer2860.8220.830

Alone it ties on returning customers but ranks first-time customers at random (0.50) — half the rows. The email/quote features carry that cold-start half (0.50 → 0.58), which is why the full model is needed even though the single feature looks "good enough" on the average.

3.5 Comparing model choices

Per the challenge's "compare model choices": we tested XGBoost (shallow, regularized) — it underperformed logistic at every t (overall 0.700 vs 0.744; see Fig 4 / the table above). No nonlinear or interaction signal to exploit, and on 130 positives the simpler linear model generalizes better and stays interpretable. We also compared one pooled model (with t as a feature) vs three per-t models: pooled tied-or-won and crucially won day-0 (it borrows strength across snapshots instead of training t=0 on cold-start-heavy rows). → ship one pooled logistic regression.

4. Conclusion — capability & features

0.74overall AUC — a genuinely useful prioritizer
2.3×top-20% lift, incl. brand-new submissions
0.76day-0 AUC — earliest = most valuable
  • The signal is simple and linear. A shallow XGBoost lost to logistic at every t — no hidden interactions to exploit.
  • The win is repeat-customer history (agent_bind_rate), which alone carries most of the overall ranking and unlocks day-0.
  • Honest limit: the model mainly "separates known-good customers from the unknown pile." ~51% of day-0 rows are first-time customers, where it can only fall back to in-submission activity (Fig 6, right). The real lever for them would be submission-static attributes (line of business, geography, premium size) — not present in this dataset.
  • Caveats: small data (130 positives), a single temporal split, and a tiny/noisy t=30 slice — trust the robust figures (~0.74 AUC, 2–3× lift), not the t=30 point estimates.

Appendix B — Overfitting check: in-sample (train) vs held-out (test)

Score the fitted model on its own training rows (in-sample) and on the held-out test, and compare.

metrictrain (in-sample)held-out testgap (train − test)
ROC-AUC0.6090.744-0.136
PR-AUC0.2410.359-0.118

No sign of overfitting — in fact the opposite. Overfitting shows train ≫ test; here the gap is negative (test beats train on every metric). Two reasons: (1) a strongly L2-regularized 5-feature linear model is low-variance — in-sample AUC (0.61) barely exceeds its CV AUC, so it isn't memorizing the train set; (2) the dominant feature agent_bind_rate accumulates over time, so the earliest-70% train window is mostly first-time agents (cold-start → feature ≈ constant, little to grip) while the later-30% test window is history-rich, where the model is genuinely more skillful.
Caveat: the negative gap isn't free generalization — the test window is also an easier, more history-rich slice; with ~130 positives and a single split, read the values as noisy. The robust takeaway is qualitative: not overfit.
