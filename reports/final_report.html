1. Data, cleaning & what to watch for

881 submissions (one per submissionId), 14.8% bind rate → imbalanced, so we use ranking metrics, not accuracy. ~17k events of 3 types (EMAIL_INBOUND / OUTBOUND / QUOTE_RECEIVED). A row is one (submission, t) pair, scored only while the submission is still open at t.

Cleaning policy (2 drops, leakage-safe):
  • 530 exact-duplicate events (same sub + second + type + chars + attach) — concentrated in high-activity subs, so they inflate the volume features that carry our signal.
  • 127 impossible pre-creation events (dated >1 day before the submission exists) — counted at every t (incl. t=0), falsely inflating day-0 activity.
Watch out for:
  • Leakage is the central risk: feature(sub, t) may use only events with eventDate ≤ createdDate + t, and never resolvedDate/label.
  • Domain quirk, NOT leakage: OUTBOUND / quotes can precede the first INBOUND — intake happens via createdDate (off-channel), carriers quote off-channel. Kept.
  • Char outliers (max 43k, p99 2.3k): kept, tamed by log1p — not deleted.
  • Censoring: by t=30 >50% have resolved; only score open-at-t submissions.

2. Features chosen (Task 1)

The signal is simple: activity volume + quote presence + customer track record. Normalized ratios, attachments, char-distribution stats, response-latency, velocity and calendar all add nothing beyond raw volume (see Appendix A).

The 4 predictive features + t (a context variable that lets one pooled model adapt across snapshots):

featurewhat it capturesleakage-safe basis
agent_bind_raterepeat-customer close-rate (smoothed)only the customer's subs resolved before this createdDate
has_quote_by_tfunnel gate — a carrier quote existsevents with d ≤ t
outbound_chars_logbroker effort (log1p of OUTBOUND chars)events with d ≤ t
n_inbound_by_tcustomer engagement (replies)events with d ≤ t
tsnapshot time (0/7/30)structural
Why agent_bind_rate matters most: agentEmail is Novella's customer (the retail agent), not the broker. A customer's prior close-rate is known at t=0 — the one strong signal available on day 0 (for returning customers), when nothing has happened on the submission yet. Adding it lifted the day-0 ranking from a coin-flip (ROC-AUC ≈0.50) to 0.76.
Note: 0.50 is the AUC floor (random ranker), not a bind rate. The base bind rate is 15% — that's the predicted probability a first-time customer gets (see the cold-start callout in §3.3).
Building agent_bind_rate — the no-history dilemma & how it updates. Two problems: a first-time agent has no history (no rate to compute), and an agent with a single prior deal that happened to bind would naively read as 100%. Solution — empirical-Bayes smoothing: blend the agent's own record with the global average, weighted by how much history they have:
rate = (prior_binds + α·g) / (prior_n + α)   α = 5 pseudo-deals, g = base rate (15%)
  • No history (prior_n = 0) → rate = g: a new customer is "assumed average until proven otherwise."
  • Self-updating: each resolved deal grows prior_n, sliding the estimate from g toward the agent's true close-rate; one fluke deal is diluted by the 5 pseudo-deals.
  • Leakage-safe: uses only the agent's deals resolved before this submission's createdDate; g is the train base rate (recomputed after the split).

Worked example (g = 15%, α = 5): new agent 0/0 → 15%  ·  one win 1/1 → (1 + 5·0.15)/6 = 29% (not 100%)  ·  established 8/20 → (8 + 5·0.15)/25 = 35% (≈ their true 40%). Thin history stays near 15%; lots of history trusts the agent.

3. Modeling (Task 3)

3.1 Preprocessing & validation

(a) Clean the data — apply the 2 drops above in-pipeline (never mutate data/). (b) Build features per (submission, t), leakage-safe. Every feature is standardized (z-scored: mean 0, sd 1) inside the model pipeline Impute(median) → StandardScaler → Logistic — fit on the training fold only. Per-feature handling:

featuretransform before scalingscaled?special handling
agent_bind_rateempirical-Bayes smoothingyessmoothed toward the TRAIN base rate; cold-start → base rate (recomputed after the split → no leakage)
outbound_chars_loglog1pyeslog first to tame the 43k-char outlier, then scale
has_quote_by_tnone (binary 0/1)yesscaled like the rest (harmless)
n_inbound_by_tnoneyes—
tnoneyesconstant within a snapshot

Why standardize: L2 regularization penalizes coefficients equally regardless of units, and the optimizer converges faster on a well-scaled loss surface; it also makes the coefficients comparable for the §2.1 ranking.

(c) Validation strategy — three-way, no peeking:
  • Temporal split: train = earliest ~70% of submissions (by createdDate); held-out test = latest ~30% (578 rows). Train-on-past / test-on-future is the honest setup for a repeat-customer signal.
  • Validation = grouped CV inside the training set: every choice (feature set, the log1p transform, regularization) is picked by 5-fold StratifiedGroupKFold, grouped by submissionId so a submission's t=0/7/30 rows never straddle a fold — no within-submission leakage. We do not carve a static validation set: with only 130 positives, CV reuses the data more efficiently.
  • The held-out test is touched once — for the final numbers below; it never informs a modeling decision.

3.2 How we measure

Precision@k is primary. Brokers work a ranked queue top-down, so the operational question is "of the top k% I'm told to work, what share actually bind?" — exactly precision@top-k (k = 20/30/50%). Lift = precision@k ÷ base rate = how much better than working at random.
Secondary: PR-AUC (precision/recall — the right summary under imbalance) and ROC-AUC (familiar, but optimistic on imbalanced data because it rewards ranking the abundant non-binders — so we don't lead with it).
The two floors differ: a random ranker has ROC-AUC = 0.50 (the AUC floor); random precision / PR-AUC = 15% (the prevalence). 0.50 is not a rate — that's why the baseline AUC floor is 0.50, not 15%.

Cold-start clarity (what t=0 really gives you): for a first-time customer at t=0, agent_bind_rate is a constant (15% base rate) → it can't rank them, and the other features are nearly empty (no quotes, ~6% have outbound; only ~44% have an early inbound). So the model scores them near 15% and the day-0 strength is really about returning customers. The score "separates known-good repeat customers from the unknown 15% pile."

3.4 Why not just the customer bind-rate?

A fair question, since agent_bind_rate alone nearly matches the full model overall. But that overall number hides a split — segment the test by whether the customer has history:

segmentnagent_bind_rate alonefull model
overall5780.7420.744
returning customer2860.8220.830
first-time (cold-start)2920.5000.578

Alone it ties on returning customers but ranks first-time customers at random (0.50) — half the rows. The email/quote features carry that cold-start half (0.50 → 0.58), which is why the full model is needed even though the single feature looks "good enough" on the average.

3.5 Comparing model choices

Per the challenge's "compare model choices": we tested XGBoost (shallow, regularized) — it underperformed logistic at every t (overall 0.696 vs 0.744; see Fig 4 / the table above). No nonlinear or interaction signal to exploit, and on 130 positives the simpler linear model generalizes better and stays interpretable. We also compared one pooled model (with t as a feature) vs three per-t models: pooled tied-or-won and crucially won day-0 (it borrows strength across snapshots instead of training t=0 on cold-start-heavy rows). → ship one pooled logistic regression.

4. Conclusion — capability & features

0.74overall AUC — a genuinely useful prioritizer
2.3×top-20% lift, incl. brand-new submissions
0.76day-0 AUC — earliest = most valuable
  • The signal is simple and linear. A shallow XGBoost lost to logistic at every t — no hidden interactions to exploit.
  • The win is repeat-customer history (agent_bind_rate), which alone carries most of the overall ranking and unlocks day-0.
  • Honest limit: the model mainly "separates known-good customers from the unknown pile." ~51% of day-0 rows are first-time customers, where it can only fall back to in-submission activity (Fig 6, right). The real lever for them would be submission-static attributes (line of business, geography, premium size) — not present in this dataset.
  • Caveats: small data (130 positives), a single temporal split, and a tiny/noisy t=30 slice — trust the robust figures (~0.74 AUC, 2–3× lift), not the t=30 point estimates.
