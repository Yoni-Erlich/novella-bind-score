Pipeline & Model Deep-Dive

How raw data becomes a bind score — every stage, the module that owns it, and the ML underneath. A study & interview companion (paired with final_report.html).

1. The pipeline at a glance

Six stages, four modules. Data flows top-to-bottom; the tag on each box is the file that owns that step.

data/*.csvRaw data

features_submissions.csv (881 subs: createdDate, resolvedDate, agentEmail, label) + features_events.csv (~17k events: type, date, char_count, attach_count).

▼
src/preprocessing.pyValidate & clean

validate() runs the full integrity battery; clean() drops 530 duplicate + 127 impossible pre-creation events → 16,564. Only removes invalid rows — never uses label/resolvedDate.

▼
src/features.pyBuild the (submission, t) panel

build_panel() makes one row per submission × t∈[0,7,30] that is still open at t (1,927 rows: 881/752/294 per t). Each row gets the 5 leakage-safe features. feature(submission_id, t) is the single-row challenge contract.

▼
src/model.pySplit → validate → fit

temporal_split(): train = earliest 70% by createdDate (≤ 2020-05-30, 1349 rows), test = latest 30% (578 rows). StratifiedGroupKFold CV inside train picks the feature set & log1p. make_model() = Impute → Scale → Logistic, fit on train.

▼
src/model.pyScore

bind_score(submission_id, t) → builds that row's features → predict_proba → P(bind). Higher = work it first.

▼
src/evaluate.pyEvaluate vs baselines

get_test_scores() scores the model + baselines once on the held-out test; precision_at_k() / per_t() produce the ranking & prioritization metrics.

Mental model: think of it as an assembly line — preprocessing cleans the parts, features stamps each submission into 3 dated snapshots, model learns the pattern and scores, evaluate is QA at the end of the line. Each station only trusts what the station before it produced.

2. Stage by stage (what happens + why)

2.1  Ingest & clean  — src/preprocessing.py

validate() runs every integrity check and reports its count even when zero (so the report shows what we look for, not just what we found). clean() applies the decided policy: drop 530 byte-identical duplicate events and 127 pre-creation events (dated >1 day before the submission exists). Char outliers are kept (handled later by log1p).

Heuristic: "clean only what is impossible or fake, not what is merely extreme." Duplicates & time-travel events are garbage; a 43k-char email is real data — transform it, don't delete it.

2.2  Feature construction  — src/features.py

The key abstraction is the panel: one row per (submission, t) for t ∈ [0, 7, 30], kept only while the submission is still open at t. So most submissions contribute up to 3 rows — three timed photographs. Each row carries the 5 features; agent_bind_rate is smoothed (empirical-Bayes) and outbound_chars is log1p-compressed.

Heuristic — the leakage rule: a feature at (sub, t) may use only what a broker could have actually seen at day t — events with eventDate ≤ createdDate + t, and for customer history, only that customer's submissions already resolved before this one was created. Never the future.
Heuristic — agent_bind_rate smoothing: "a new customer is assumed average until proven otherwise." With 0 prior deals you get the base rate; each resolved deal pulls the estimate toward that customer's true close-rate. (That's why first-timers all share one value — the cold-start problem.)

2.3  Split & validate  — src/model.py

temporal_split() trains on the earliest ~70% of submissions and tests on the latest ~30%. Model choices are made by StratifiedGroupKFold cross-validation inside the training set, grouped by submissionId. The test set is scored once, at the end.

Heuristic — temporal split: "train on the past, predict the future" — the only honest test for a feature like customer history that accumulates over time. A random split would let the model peek at later deals from the same era.
Heuristic — grouped CV: keep a submission's 3 snapshots (t=0/7/30) together in one fold. Split them and you'd train on t=0 and validate on t=30 of the same submission — testing on a near-copy of training, which flatters the score.

2.4  The model pipeline  — src/model.py

make_pipeline(
    SimpleImputer(strategy="median"),   # fill gaps
    StandardScaler(),                    # normalize every feature -> mean 0, sd 1
    LogisticRegression(class_weight="balanced", C=1.0),  # L2-regularized
)

One pooled logistic regression with t as a feature (not 3 separate models). Everything is fit inside train/the fold only, so test statistics never leak into the scaler or the imputer.

Heuristic — class_weight="balanced": only 15% of rows bind, so a lazy model would predict "never binds" and be 85% accurate but useless. Balancing up-weights the rare "sold" class in the loss so the model actually has to separate them. (§4 covers why this matters more than raw accuracy.)

2.5  Score & evaluate  — src/model.py + src/evaluate.py

bind_score(submission_id, t) rebuilds that one row's features and returns predict_proba = P(bind) — the deliverable. evaluate.py scores the model and every baseline once on the held-out test and computes precision@k / lift / per-t AUC.

3. Deep-dive: why normalize features?

Logistic regression's fit quality doesn't care about units in theory — but the optimizer that finds the weights very much does. Feature scale sets the shape of the loss surface the optimizer must descend.

The four ways unscaled features interfere

  • Ill-conditioned surface. Different scales → very different curvature per axis → the Hessian's condition number (κ = λmax/λmin) blows up. Gradient-descent convergence speed degrades directly with κ.
  • One learning rate can't serve all axes. The step must be small enough to stay stable on the steep axis — but then it's far too small on the flat axis. Result: zig-zag across the valley, crawl along it. With a fixed max_iter, it can stop before converging (sklearn's ConvergenceWarning) and return an under-fit model.
  • Numerical saturation. A raw value like outbound_chars = 43,650 makes wᵀx enormous → the sigmoid saturates (≈0 or 1) → its gradient ≈ 0 → no learning signal; you can also overflow exp.
  • Regularization is distorted. L2 penalizes Σwⱼ² equally, ignoring units. A large-scale feature needs only a tiny weight (so L2 barely touches it); a small-scale feature needs a big weight (so L2 over-shrinks it). Which features survive the penalty would depend on arbitrary unit choices — unless you standardize first.
One-line takeaway: standardizing turns a long skewed valley into a round bowl → the optimizer converges fast and reliably, and the L2 penalty treats every feature fairly. (Trees, e.g. our XGBoost comparison, are scale-invariant and need none of this.) In our model outbound_chars also gets log1p first, to pull in its long tail before scaling.

4. Deep-dive: logistic regression (interview prep)

The model

Logistic regression models the log-odds of the positive class as a linear function of the features:

logit(p) = ln( p / (1−p) ) = wᵀx + b        ⟹        p = σ(wᵀx + b) = 1 / (1 + e^−(wᵀx+b))

So it's a linear model wrapped in a sigmoid that squashes the score into a probability in (0, 1). The decision boundary is linear in x (a hyperplane where wᵀx + b = 0, i.e. p = 0.5).

The loss function — binary cross-entropy (log-loss)

Derived from maximum likelihood: each label is Bernoulli(pi), so the likelihood is ∏ piyi(1−pi)1−yi. Maximizing it = minimizing the negative log-likelihood, the cross-entropy:

L(w) = − Σᵢ [ yᵢ·ln(pᵢ) + (1−yᵢ)·ln(1−pᵢ) ]   +   λ‖w‖²   (L2 term)
Why log-loss and not MSE? Paired with the sigmoid, squared error is non-convex in w (local minima) and its gradient vanishes when the sigmoid saturates — so training stalls on confident-wrong examples. Cross-entropy is convex (with L2, strictly convex → unique global optimum) and its gradient stays strong exactly where the model is most wrong.

How it's optimized

The gradient has a famously clean form — "prediction error times input":

∇w L = Xᵀ (p − y)        (p − y is the residual; no closed-form solution → solved iteratively)

sklearn's default solver (L-BFGS) is a quasi-Newton method that uses curvature to converge fast — but, per §3, only when the features are scaled. C = 1/λ controls regularization strength (smaller C = stronger shrink).

Interpreting it

  • Raw coefficient: ewj is the odds ratio — the multiplicative change in odds of binding per one-unit increase in feature j.
  • Standardized coefficient: after StandardScaler, |wj| is comparable across features → that's our Task-2 importance ranking (effect per 1 standard deviation).

Assumptions, strengths, weaknesses

AssumesStrengthsWeaknesses
Linear in the log-odds · independent observations · limited multicollinearity · features scaled (for the solver/penalty) Interpretable · well-calibrated probabilities · fast · few parameters · strong baseline · convex (no local minima) Only a linear boundary (needs feature engineering for interactions) · sensitive to scaling/outliers · unstable coefficients under collinearity (L2 mitigates)

Why it fits this problem

The signal here is simple and roughly linear (a shallow XGBoost lost at every t — no interactions to exploit), the data is small (130 positives → a low-variance linear model resists overfitting), and brokers want an interpretable, calibrated score. Logistic regression is the right tool; XGBoost was the comparison, not the ship.

5. Interview cheat-sheet

Q. Why not just use accuracy?

Only 15% bind — predicting "never" scores 85% accuracy yet ranks nothing. We use ranking metrics (ROC/PR-AUC) and the operational one, precision@top-k (brokers work a ranked queue).

Q. How do you handle the class imbalance?

class_weight="balanced" up-weights the rare class in the loss; we judge on PR-AUC / precision@k, not accuracy. (It nudges predicted probabilities up — fine, since we only need the ranking.)

Q. How do you prevent leakage?

Features obey eventDate ≤ createdDate + t; customer history uses only prior-resolved deals; temporal train/past → test/future split; scaler/imputer fit inside the fold; grouped CV keeps a submission's rows together.

Q. Why logistic over XGBoost here?

Tested both — XGBoost lost at every t. No nonlinear/interaction signal to exploit, and on 130 positives the simpler model generalizes better and stays interpretable.

Q. What does standardizing change?

Nothing about the ideal fit, everything about reaching it: it conditions the loss surface for the optimizer and lets the L2 penalty treat features fairly (§3). It also makes coefficients comparable for ranking.

Q. Biggest limitation?

Cold-start: ~half of day-0 rows are first-time customers with no history, where the model falls back near the base rate. The fix would be submission-static attributes (line of business, geography, premium) — not in this data.
